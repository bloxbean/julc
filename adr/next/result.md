│ Libraries That Cannot Be Migrated (4 of 11)                                                                                                                                                                            │
│                                                                                                                                                                                                                        │
│ ListsLib — PlutusData.ListData is the fundamental type                                                                                                                                                                 │
│                                                                                                                                                                                                                        │
│ ListData IS the on-chain list type. There is no java.util.List<T> equivalent at UPLC level — the compiler maps PlutusData.ListData to UPLC built-in list operations (headList, tailList, nullList, mkCons). Changing   │
│ to java.util.List would require the compiler to emit completely different UPLC code, and would lose the ability to work with heterogeneous UPLC lists.                                                                 │
│                                                                                                                                                                                                                        │
│ The off-chain stub does have head(List<T>) / tail(List<T>) overloads for testing convenience, but these can't replace the on-chain ListData API.                                                                       │
│                                                                                                                                                                                                                        │
│ MapLib — PlutusData.MapData is the fundamental type                                                                                                                                                                    │
│                                                                                                                                                                                                                        │
│ Same reasoning. MapData IS the on-chain map (association list). The operations (unMapData, mapData, fstPair, sndPair) are UPLC built-ins that operate on this specific representation. There's no java.util.Map<K,V>   │
│ equivalent at UPLC level.                                                                                                                                                                                              │
│                                                                                                                                                                                                                        │
│ ContextsLib — manual ConstrData field extraction                                                                                                                                                                       │
│                                                                                                                                                                                                                        │
│ This library exists to provide low-level field extraction from ScriptContext/TxInfo encoded as ConstrData. It manually calls constrFields() + headList/tailList by positional index. While the off-chain stub uses     │
│ TxInfo/ScriptContext ledger types, the on-chain code can't use those types because:                                                                                                                                    │
│ 1. The methods construct new ConstrData values (e.g., constrData(0, ...) for Optional results) — the compiler doesn't support record construction, only pattern matching                                               │
│ 2. The methods do raw positional field extraction (listIndex(fields, 8)) which is the whole purpose of this library                                                                                                    │
│ 3. Users who want typed access already get it for free via the compiler's record type support for TxInfo/ScriptContext — ContextsLib is the escape hatch for low-level access                                          │
│                                                                                                                                                                                                                        │
│ IntervalLib — constructs ConstrData from scratch                                                                                                                                                                       │
│                                                                                                                                                                                                                        │
│ This library both creates and destructures interval representations using Builtins.constrData(tag, fields). Methods like between(), after(), before() build Interval ConstrData manually. The compiler can destructure │
│  ledger records but cannot construct them via new Interval(...). The off-chain stub uses the Interval ledger type because it can instantiate Java objects.
