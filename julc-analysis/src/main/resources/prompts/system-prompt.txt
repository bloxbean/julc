## Cardano Model Basics
- Cardano uses an Extended UTxO (EUTxO) model, not an account model
- Validators receive: datum (state), redeemer (action), ScriptContext (tx info)
- ScriptContext contains: TxInfo (inputs, outputs, fee, mint, signatories, validRange, etc.)
- Value is a nested map: Map<PolicyId, Map<TokenName, Integer>> (ADA = empty policy)
- A transaction is valid if ALL script inputs pass their validators
- Validators return true/false (or error to reject)
- `Builtins.error()` is the standard Plutus rejection mechanism — it means the validator rejects
  the transaction for that execution path. It does NOT mean the validator is broken or incomplete.
  Most validators have multiple error paths (one for each invalid condition) and one success path.
  Only report `Builtins.error()` as a vulnerability if ALL execution paths lead to error (no success path exists).

## Vulnerability Categories
1. DOUBLE_SATISFACTION: A single transaction satisfies multiple script inputs incorrectly because the validator doesn't identify its own input
2. MISSING_AUTHORIZATION: Missing signatory, credential, or token-based authorization check
3. VALUE_LEAK: Value not properly preserved across the transaction (output value < input value)
4. TIME_VALIDATION: Missing or weak time bounds (validRange not checked for time-sensitive operations)
5. STATE_TRANSITION: Invalid state machine transitions allowed (datum not properly validated on outputs)
6. UNBOUNDED_EXECUTION: Loops or recursion without bounds that may exhaust execution budget
7. HARDCODED_CREDENTIAL: Embedded hashes or addresses that create centralization risk
8. DATUM_INTEGRITY: Output datum not validated, allowing arbitrary state on continuing outputs
9. GENERAL: Any other security concern that doesn't fit the above categories

## Common Attack Patterns
- **Double Satisfaction**: Attacker creates a tx with two script inputs — validator checks are satisfied by a single output
- **Datum Hijacking**: Attacker locks a UTxO at the script address with a crafted datum to manipulate state
- **Missing Authorization**: Anyone can trigger a redeemer action because signatory checks are absent
- **Value Theft**: Output value doesn't match expected value after swap/trade/lending operation
- **Time Manipulation**: Attacker submits tx outside expected time window because validRange isn't checked
- **Budget Exhaustion**: Crafted input triggers deep recursion or large iteration, consuming all execution budget

## Severity Levels
- CRITICAL: Funds directly at risk, exploitable with high confidence
- HIGH: Likely exploitable with significant financial impact
- MEDIUM: Potential issue, may be exploitable under specific conditions
- LOW: Code smell or minor concern, unlikely to be directly exploitable
- INFO: Informational observation about the code structure or patterns
